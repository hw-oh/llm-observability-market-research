# LLM Observability — 製品詳細
**日付**: 2026-02-13 | **モデル**: google/gemini-3-pro-preview

### W&B Weave

**概要**: W&B Weaveは、広範な Weights & Biases エコシステムと深く統合された、開発者中心の LLM Observability および Eval プラットフォームです。プログラムによる Tracing、マルチモーダル Eval（オーディオを含む）、エンタープライズグレードのインフラストラクチャに優れていますが、ノーコードのプロンプト管理やプロキシベースの統合よりも、コードファーストのワークフローを優先しています。

**強み**:
- 強力なマルチモーダル Tracing および Eval 機能（テキスト、オーディオ、ビデオ）
- ライフサイクル全体の管理を可能にする W&B Experiments との深い統合
- セルフホストやコンプライアンスオプションを備えた堅牢なエンタープライズインフラ
- カスタム Scoring と動的リーダーボードを備えた柔軟な Eval フレームワーク
- Model Context Protocol (MCP) Tracing のネイティブサポート

**弱み**:
- ノーコードのプロンプト管理（CMS）機能の欠如
- ゼロコード統合のためのプロキシ/ゲートウェイモードがない
- 特化した RAG リトリーバル可視化の欠如
- 組み込みのキュー管理がない手動アノテーションワークフロー
- エージェントの可視化がトレースツリーに限定され、上位レベルの DAG ビューがない

**最新のアップデート**:
- Audio Monitors: オーディオ出力用のオンライン Eval モニター。LLM Judge がテキストと並んで MP3/WAV ファイルを評価可能に。（2026-02-01）
- Dynamic Leaderboards: フィルタやメトリクスの永続的なカスタマイズが可能な、Eval における自動生成リーダーボード。（2026-01-29）
- Custom LoRAs in Playground: Weave Playground で直接、カスタム Fine-tuning された LoRA 重みをテストおよび比較するためのサポート。（2026-01-16）

### LangSmith

**概要**: LangSmith は、LangChain エコシステムの深い Observability、Eval、およびコラボレーションに特化した、LLM アプリケーション向けの包括的な DevOps プラットフォームです。複雑なエージェントワークフローのきめ細かな Tracing と、堅牢なデータセット管理およびヒューマンインザループのアノテーションツールを組み合わせていますが、Guardrails や高度な Eval についてはコード中心の設定に大きく依存しています。

**強み**:
- 複雑なエージェントのシームレスな自動インスツルメンテーションを実現する、LangChain との深いネイティブ統合。
- 自動化されたコード/LLM Scoring と手動アノテーションキューを組み合わせた包括的な Eval ワークフロー。
- マルチクラウド SaaS、BYOC、セルフホスト型 Kubernetes を含む、強力なエンタープライズデプロイオプション。
- ネストされたスパン、Streaming レスポンス、ツール使用のための高度な Tracing 機能。

**弱み**:
- 外部ライブラリの統合なしでは、セーフティ（ハルシネーション、ジェイルブレイク）のためのネイティブな Guardrails が不足している。
- LLM-as-a-judge エバリュエーターを設定するための「ノーコード」ウィザードがなく、プロンプトエンジニアリングが必要。
- Go SDK およびプロキシベースの Tracing モードがないため、一部のスタックで統合の柔軟性が制限される。

**最新のアップデート**:
- Customize trace previews: LangSmith UI でのトレースプレビューのレンダリング方法をカスタマイズする機能。（2026-02-06）
- LangSmith Self-Hosted v0.13: セルフホスト型エンタープライズデプロイオプションの新しいバージョンリリース。（2026-01-16）

### Langfuse

**概要**: Langfuse は、オープンソースで開発者中心の LLM エンジニアリングプラットフォームであり、堅牢な Observability と包括的な Eval 機能を組み合わせています。複雑な RAG やエージェントワークフローの Tracing に優れ、セルフホストや SaaS を含む柔軟なデプロイオプションを提供すると同時に、プロンプト管理や実験のための統合ツールを提供します。

**強み**:
- 完全にオープンソースでセルフホスト可能であり、最大限のデータ制御を提供
- LangChain や LlamaIndex などの人気のあるフレームワークとの強力な統合
- LLM-as-a-Judge や手動アノテーションキューを含む包括的な Eval スイート
- 迅速なイテレーションのための統合されたプロンプト管理と Playground
- SaaS からエアギャップ環境の VPC まで対応する柔軟なデプロイオプション

**弱み**:
- ネイティブな Guardrails および Policy-as-code 適用の欠如
- マルチモーダル Tracing（画像/オーディオ）のサポートなし
- A/B テストはサポートしているが、専用の回帰テスト UI がない
- 公式の Go SDK がない
- Fine-tuning データパイプライン統合の欠如

**最新のアップデート**:
- LLM-as-a-Judge on Observations: 個別の Observation に対して直接 LLM-as-a-judge Eval を実行するサポートを追加。（2026-02-13）
- Single Observation Evals: 単一の Observation に対する Eval ワークフローを有効化。（2026-02-10）
- Thinking/Reasoning Rendering: トレース詳細に思考（Thinking）と推論（Reasoning）部分のレンダリングを追加し、Chain-of-Thought の可視性を向上。（2026-01-28）
- Org Audit Log Viewer: 組織レベルの監査ログ用ビューアーを導入。（2026-01-28）
- Inline Trace Comments: トレース内の入出力データの一部に対してインラインでコメントを追加可能に。（2026-01-20）

### Braintrust

**概要**: Braintrust は、Eval、プロンプト管理、および Observability を単一のワークフローに緊密に統合したエンタープライズグレードの AI 開発プラットフォームです。「Eval 駆動開発」に重点を置いているのが特徴で、回帰テスト、データセットキュレーション、およびデータプライバシーのためのハイブリッドセルフホストデプロイのための堅牢なツールを提供します。開発ライフサイクルとカスタム Scoring には優れていますが、現在、複雑なエージェントグラフの専用可視化や、専用のリアルタイム Guardrails 適用機能は不足しています。

**強み**:
- カスタム Scoring と回帰テストを備えた包括的な Eval フレームワーク
- プロンプト CMS や Playground を含む強力な開発ライフサイクルツール
- データが顧客のクラウド内に留まることを保証するハイブリッドデプロイモデル
- Python、TypeScript、および Go 向けの堅牢な SDK サポート
- 統合されたデータセット管理およびキュレーションワークフロー

**弱み**:
- 高度なエージェント実行グラフの可視化の欠如
- PII やジェイルブレイク検出のための専用のリアルタイム Guardrails がない
- CI/CD パイプラインやデータウェアハウス向けの構築済み統合が限定的
- オープンソース版やセルフホストのコミュニティエディションがない
- 埋め込みやベクトルリトリーバルのための専用の可視化が欠けている

**最新のアップデート**:
- Sub-agent nesting for Claude Agent: Claude Agent SDK ラッパーにサブエージェントのネストのサポートを追加し、複雑なエージェントワークフローの Tracing を改善。（2026-02-12）
- Classifications Field: SDK に新しい「Classifications」フィールドを導入。トレースのメタデータタグ付けや分類の強化を目的としている。（2026-01-31）
- Eval Cache Control: Eval 中およびスパンエクスポート後のキャッシュをオフにするオプションを追加し、実験の再現性をより細かく制御可能に。（2026-01-29）
- Python Trace Scoring: Python でのトレース Scoring の新しい候補実装。プログラムによる Eval 機能を強化。（2026-01-21）
- Review Span Type: SDK に特定の「review」スパンタイプを追加し、トレース内での人間によるレビューステップの分類を容易に。（2026-01-15）

### MLflow

**概要**: MLflow は、成熟したオープンソースの MLOps プラットフォームであり、堅牢な Tracing、Eval、および実験トラッキング機能により、GenAI Observability 分野への拡張に成功しています。コード中心のワークフロー、カスタムメトリクス、および自動インスツルメンテーションによる本番モニタリングに優れていますが、LLM ネイティブな特化型ツールと比較すると、専用のエージェント可視化や非技術者向けのコラボレーション機能はまだ進化の途上にあります。最近のアップデートでは、プロンプト管理とマルチワークスペース機能が強化され、エンジニアリングチーム向けの多才な標準ツールとしての地位を固めています。

**強み**:
- 膨大なコミュニティの採用実績を持つ、業界標準の実験トラッキングとモデルレジストリ。
- 主要な LLM ライブラリ（LangChain、LlamaIndex、OpenAI）に対する強力な自動インスツルメンテーション。
- ローカルのオープンソースからマネージドなエンタープライズ SaaS まで対応する柔軟なデプロイオプション。
- コードベースの Scoring とデータセットをサポートする堅牢なカスタム Eval 機能。
- 広範なデータエンジニアリングエコシステム（Databricks、Spark）との深い統合。

**弱み**:
- 実行グラフやセッションリプレイのような専用のエージェント可視化ツールが不足している。
- プロンプトエンジニアリングとテストのためのネイティブなインタラクティブ Playground がない。
- 特化型の競合他社と比較して、非技術者向けのコラボレーション機能（CMS、ノーコードウィザード）が限定的。
- 組み込みのコスト分析および属性把握機能が欠けている。
- RBAC や監査ログなどのエンタープライズセキュリティ機能は、オープンソース版では制限されている。

**最新のアップデート**:
- Organization Support: MLflow Tracking Server におけるマルチワークスペース環境のサポート。異なるワークスペース間での実験の整理が可能に。（2026-02-12）
- MLflow Assistant: MLflow UI 内で直接問題を特定、診断、修正するのを支援する、Claude Code を活用した製品内チャットボット。（2026-01-29）

### Arize Phoenix

**概要**: Arize Phoenix は、OpenTelemetry 標準に基づいて構築されたオープンソースの AI Observability および Eval プラットフォームであり、開発と本番のギャップを埋めるように設計されています。RAG アプリケーションの深い Tracing、LLM-as-a-judge Eval ワークフロー、および主要な Python フレームワーク向けのシームレスな自動インスツルメンテーションに優れていますが、高度な本番モニタリングやチームコラボレーション機能については、より広範な Arize エコシステムに依存しています。

**強み**:
- 深い RAG 可視化を備えた、包括的な OpenTelemetry ベースの Tracing
- 構築済みおよびカスタムのエバリュエーターを備えた強力な LLM-as-a-judge 機能
- LangChain や LlamaIndex などの主要フレームワーク向けのシームレスな自動インスツルメンテーション
- オープンソースのセルフホストおよび SaaS を含む柔軟なデプロイオプション

**弱み**:
- ネイティブなプロンプト管理（CMS）およびバージョン管理機能の欠如
- 限定的なコスト分析および属性把握機能
- マルチモーダル Tracing や Streaming レスポンス可視化に関するドキュメント化されたサポートがない
- 統合を簡素化するためのプロキシ/ゲートウェイモードの欠如

**最新のアップデート**:
- Claude Opus 4.6 Support: Playground に Claude Opus 4.6 モデルのサポートを追加。（2026-02-09）
- Tool Selection Evaluator: ライブラリに不足していた tool_selection エバリュエーターを追加。（2026-02-06）
- Faithfulness Evaluator: FaithfulnessEvaluator を導入し、HallucinationEvaluator を非推奨化。（2026-02-02）
- Tool Invocation Accuracy Metric: ツール呼び出しの精度を追跡するための新しいメトリクスを追加。（2026-01-27）
- Cursor Rule for Metrics: 新しい組み込みメトリクス（LLM 分類エバリュエーター）を作成するための Cursor ルールを追加。（2026-01-21）
