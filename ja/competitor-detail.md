# LLM Observability — 製品詳細
**日付**: 2026-02-25 | **モデル**: google/gemini-3-pro-preview

### W&B Weave

**概要**: W&B Weaveは、Weights & Biasesの強力な基盤を活用し、軽量なTracingツールから包括的なLLM opsプラットフォームへと急速に進化しました。2026年初頭の最新アップデートでは、オーディオモニター、動的リーダーボード、エンタープライズグレードのTrace分析が導入され、機能のギャップが大幅に解消されました。堅牢なマルチモーダルサポート、W&Bのトレーニング/Fine-tuningエコシステムとの緊密な連携、より強力なGuardrails（PII、ハルシネーション）を備えており、実験から本番環境までエンドツーエンドの可視性を必要とする開発者にとってトップティアの選択肢となっています。スタンドアロンのプロキシゲートウェイや高度なEmbeddingの可視化機能は欠けていますが、開発者中心のSDKと、Evalおよびモニタリングのためのシームレスなワークフローに強みがあります。

**強み**:
- W&B Experimentsとのシームレスな統合により、トレーニングから本番への移行がスムーズ。
- 新しいオーディオモニタリング機能を含む、ネイティブなマルチモーダルサポート。
- 強力なエンタープライズコンプライアンス（SOC2/HIPAA）と柔軟なデプロイメント（SaaS/オンプレミス）。
- コードベースおよびLLMベースのScoringの両方をサポートする柔軟なEvalフレームワーク。
- 主要なデータウェアハウス（Snowflake、BigQuery）への堅牢なデータエクスポート機能。

**弱み**:
- SDKによる実装なしでモデルルーティングを行うための、スタンドアロンのAPIプロキシ/ゲートウェイがない。
- アノテーションワークフローが手動であり、大規模チーム向けの高度なキュー管理機能が不足している。
- セマンティッククラスタリングのためのネイティブなEmbedding空間の可視化機能がない。
- プロンプトの自動最適化や候補生成をサポートしていない。

**最新のアップデート**:
- Trace分析オーバーレイ: リクエスト数、レイテンシのパーセンタイル、トークン使用量、コストを表示するプロジェクト概要。（2026-02-23）
- Trace比較サマリー: 集約されたツール使用状況、Scoring、コストを伴うTrace比較のためのフラット化されたビュー。（2026-02-23）
- オーディオモニター: LLM Judgeを使用してオーディオ入力を監視・判定するモニターの作成をサポート。（2026-02-01）
- 動的リーダーボード: Evalから自動生成されるリーダーボード。永続的なカスタマイズとCSVエクスポートが可能。（2026-01-29）

### LangSmith

**概要**: LangSmithは、LangChainエコシステムと深く統合された特化型のObservabilityおよびEvalプラットフォームとしての役割を維持しつつ、OpenTelemetryを通じて一般的なLLMエンジニアリングへのサポートを拡大しています。このプラットフォームは、複雑なAgentワークフローの可視化に優れており、ネストされたスパン、ツール使用、検索ステップのきめ細かなTracingを提供します。最近の開発では、Agent実行のためのサンドボックス環境の強化や、カスタマイズ可能なTraceビューによる開発者の利便性向上に重点が置かれています。アノテーションキューやペア比較を含む堅牢な「ヒューマン・イン・ザ・ループ」Eval機能で差別化を図る一方、エンタープライズ向けのセルフホストデプロイオプションも提供しています。

**強み**:
- LangChainおよびLangGraphとの深いネイティブ統合による、Agentの軌跡の可視化。
- アノテーションキューと競合管理を備えた堅牢な「ヒューマン・イン・ザ・ループ」ワークフロー。
- SaaS、ハイブリッド、セルフホストVPCを含む柔軟なデプロイモデル。
- 複雑でネストされたAgentループ専用に設計された包括的なTracing。
- バージョン管理やテスト用Playgroundと統合された強力なプロンプト管理。

**弱み**:
- プロプライエタリなプラットフォームであり、オープンソースのコミュニティ版がない。
- ネイティブなGateway/Proxyサービスがないため、SDKまたはOTELへの依存が必要。
- ネイティブなEmbedding空間の可視化ツール（UMAP/t-SNEなど）がない。
- 外部ウェアハウス（Snowflake/BigQuery）へのデータ同期のための自動統合が限定的。
- プロキシベースのソリューションと比較して、LangChain以外のフレームワークでは個別のSDK/ラッパーへの依存が生じる。

**最新のアップデート**:
- サンドボックスの例外タイプとプランミング: Agentサンドボックスでのエラーハンドリングを改善するため、例外タイプとクライアントプランミングを追加。（2026-02-21）
- Google Gen AIラッパーのエクスポート: Google Gen AIラッパーのエクスポート機能と非OTELラッパーのサポートを追加。（2026-02-02）
- Traceプレビューのカスタマイズ: LangSmith UI内でのTraceプレビューの表示方法をカスタマイズする機能。（2026-02-06）

### Langfuse

**概要**: Langfuseは、深いObservability（Tracing/デバッグ）と堅牢なプロンプト管理CMSによって差別化を図り、主要なオープンソースLLMエンジニアリングプラットフォームとしての地位を固めました。2026年初頭の最新アップデート（v3.149-v3.155）では、きめ細かなEval機能（Single Span Evals、Observationに対するLLM-as-a-judge）とパフォーマンス向上（イベントベースのテーブル、ブルームフィルタ）への強いシフトが見られます。開発者体験（DX）、SDK統合、エンタープライズ対応（RBAC、SSO、SOC 2）に優れていますが、Guardrailsについてはネイティブなファイアウォールプロキシを提供するのではなく統合に依存しており、現在は高度なEmbedding空間の可視化機能が不足しています。

**強み**:
- オープンソース & セルフホスト可能: 完全なデータコントロールと柔軟なデプロイオプション（Docker/Cloud）を提供。
- 統合されたプロンプト管理: プラットフォーム内で直接プロンプトのバージョニングや編集ができる強力なCMS機能。
- 包括的なEvalスイート: 自動化されたLLM-as-a-judge、手動アノテーションキュー、回帰テストを統合。
- 開発者ファーストの統合: 自動実装（Auto-instrumentation）とOTEL互換性を備えた優れたSDKサポート。
- エンタープライズ対応: SOC 2/HIPAA準拠、SSO、RBACサポートをドキュメント化。

**弱み**:
- ネイティブプロキシ/ゲートウェイなし: リアルタイムのトラフィック遮断やルーティングのための専用プロキシがない。
- 限定的なGuardrails: ネイティブなファイアウォールエンジンの代わりに、安全性のためのブロッキングを外部統合に依存している。
- 可視化の深さ: 専門的なリサーチツールと比較して、高度なEmbedding空間の可視化（UMAP/t-SNE）が欠けている。
- Streamingサポートの詳細: サポートはされていますが、Streamingロジック（Time to First Token）に関する詳細なAnalyticsは、一般的なTracingほど強調されていません。

**最新のアップデート**:
- Single Span Evals: 個別のスパンに対してEvalを実行する機能（Beta）を導入し、品質チェックの粒度を向上。（2026-02-15）
- Observationに対するLLM-as-a-judge: Trace内の特定のObservationをターゲットにして、より的を絞った自動フィードバックを行うようLLM-as-a-judge機能を拡張。（2026-02-10）
- イベントベースのTraceテーブル: パフォーマンスとフィルタリングを改善するため、イベントベースのアーキテクチャを利用するようにTrace/Observationテーブルを最適化。（2026-02-05）
- ブルームフィルタインデックス: user_idおよびsession_idのクエリにブルームフィルタインデックスを追加し、大規模データセットでの検索を大幅に高速化。（2026-02-20）

### Braintrust

**概要**: Braintrustは、ソフトウェア開発ライフサイクルと密接に統合された、開発者中心のEvalおよびObservabilityプラットフォームです。プロンプトエンジニアリングとScorer作成を加速させるAIアシスタント「Loop」や、自動回帰テストのための堅牢なCI/CD統合で差別化を図っています。最近のアップデートでは、スレッディングや分類のためのSDK機能の深化を通じてAgentワークフローのサポートを強化し、セキュリティとキャッシュのための専用AI Proxyを導入しました。

**強み**:
- プロンプトとScorerの自動最適化のためのLoop AIアシスタント。
- 自動回帰テストのためのシームレスなCI/CD統合。
- 本番環境のTraceやデータセットに直接接続する統合Playground。
- 「LLM-as-a-judge」ウィザードを備えたEvalファーストのアプローチ。
- JS/TypeScript AIエコシステム（Vercel AI SDK）への強力なサポート。

**弱み**:
- PIIの自動検出やマスキング機能が組み込まれていない。
- データ探索のためのEmbedding空間の可視化（UMAPなど）が不足している。
- トピックブロッキングやジェイルブレイク防止のためのネイティブなリアルタイムGuardrailsがない。
- 統合されたFine-tuningパイプラインがない（最初にデータをエクスポートする必要がある）。

**最新のアップデート**:
- 公開Span Nameプロパティ: Traceの識別を容易にするため、Python SDKのSpanインターフェースに公開名プロパティを追加。（2026-02-12）
- Pythonスレッド取得: Python SDK内でスレッドコンテキストを直接取得する新機能。（2026-02-12）
- 分類フィールド: より豊かなデータラベリングのため、Python SDKでの分類フィールドのサポートを導入。（2026-01-31）
- Evalキャッシュ制御: 最新の結果を保証するため、Eval中にキャッシュを明示的にオフにするオプションを追加。（2026-01-29）
- 実験タグ: 整理を容易にするため、実験作成時にタグを渡せるように変更。（2026-02-25）

### MLflow

**概要**: MLflowは、機械学習ライフサイクルのデファクトスタンダードなオープンソースであり、3.xリリースでGenAI分野へ大幅に拡張されました。OpenTelemetry準拠の堅牢なTracing、包括的な実験トラッキング、強力なプロンプト管理を提供します。最新アップデート（v3.10）では、エンタープライズの分離ニーズに対応するため、マルチワークスペース環境の組織レベルのサポートが導入されました。Python中心の開発、統合、データ主権においては卓越していますが、ネイティブなGuardrails、高度なコスト分析、共同アノテーションワークフローの面では、専門的な商用ベンダーに後れを取っています。

**強み**:
- 20以上のライブラリに対応した1行の自動実装を備えた、OpenTelemetryネイティブなTracing。
- 比類のないオープンソースコミュニティのサポートとデータ主権。
- バージョニングとモデル構成を備えた包括的なプロンプトレジストリ。
- DSPyサポートとカスタムJudgeを備えた強力なEvalフレームワーク。
- エンタープライズの分離のための新しいマルチワークスペース組織サポート。

**弱み**:
- ネイティブなトピックブロッキングやジェイルブレイクGuardrailsの欠如。
- 専用のコスト分析や財務属性機能がない。
- 限定的な共同アノテーション機能（キューがない）。
- SDKサポートが主にPythonとJavaScriptに集中している。
- 高度なEmbedding空間の可視化機能がない。

**最新のアップデート**:
- MLflow Tracking Serverでの組織サポート: 実験とモデルの論理的な分離と整理を可能にするマルチワークスペース環境をサポート。（2026-02-20）
- MLflow Assistant: UI内で直接問題を特定、診断、修正するための、Claude Codeをバックエンドとした製品内チャットボット。（2026-01-29）

### Arize Phoenix

**概要**: Arize Phoenixは、OpenInferenceおよびOpenTelemetry標準に基づいて構築された、主要なオープンソースAI ObservabilityおよびEvalプラットフォームです。現在のバージョン13.3.0では、Embeddingクラスターや検索インスペクションなどの深い可視化ツールを使用して、複雑なLLMアプリケーション、RAGパイプライン、AgentワークフローのTracingに優れています。LLM-as-a-judgeや回帰テストを含む堅牢なEval機能を提供していますが、リアルタイムのプロキシベースのGuardrailsやブロッキングよりも、Observabilityと分析に重点を置いています。v13.0以降の最新アップデートでは、簡潔さのエバリュエーター、ネイティブなModel Context Protocol (MCP) 統合、プロンプト編集における開発者体験の向上が導入されました。

**強み**:
- ネイティブなOpenTelemetry統合により、ベンダーに依存しないTracingと幅広い互換性を確保。
- Embedding空間（UMAP）およびRAGドキュメント検索のための最高クラスの可視化。
- ツール呼び出しや中間状態のTracingを含む、複雑なAgentワークフローへの強力なサポート。
- 完全にオープンソースのコアにより、ローカルデプロイと完全なデータ主権が可能。
- カスタムPython Scorerと厳格な回帰テストを可能にする包括的なEvalスイート。

**弱み**:
- トラフィックの遮断やキャッシュのためのリアルタイムプロキシ/ゲートウェイ機能がない。
- PIIマスキングやアクティブなジェイルブレイクブロッキングのための組み込みGuardrailsがない。
- マルチモーダルTracing（画像/音声）がネイティブにサポートまたはドキュメント化されていない。
- TraceをFine-tuningジョブに直接エクスポートするための統合パイプラインがない。
- 主要な言語サポートがPython/JSに集中しており、他の言語は手動のOTEL実装が必要。

**最新のアップデート**:
- 簡潔さ分類エバリュエーター: LLM出力の簡潔さを評価するための新しいエバリュエーターを追加。（2026-02-20）
- AWS Bedrockクロスリージョン設定: AWS Bedrockクロスリージョン推論のモデルプレフィックス設定オプション。（2026-02-19）
- エバリュエーター詳細へのモデル情報追加: エバリュエーター詳細ビューにモデル情報を直接表示し、可視性を向上。（2026-02-18）
- LLM Evalプロンプトエディタのオートコンプリート: Eval設定を容易にするため、プロンプトエディタにオートコンプリート機能を追加。（2026-02-13）
- ツールレスポンス処理エバリュエーター: モデルがツールのレスポンスをどのように処理するかを評価するための新しいテンプレート。（2026-02-13）
