# W&B Weave — 製品詳細
**日付**: 2026-02-11 | **モデル**: google/gemini-3-pro-preview

[← ホーム](./) · [詳細比較](./comparison)

### Weave

**概要**: Weave は、LLM エンジニアリングにおける「科学的」な選択肢としての地位を確立しており、オブザーバビリティ（可観測性）を体系的な評価およびモデルの改善と密接に結びつけています。モニタリングのみに焦点を当てた競合（Helicone）やデプロイメント重視の競合（LangSmith）とは異なり、Weave は Weights & Biases の実績を活かし、プロダクションログをトレーニングデータセットやファインチューニングジョブに直接接続する統合された「データフライホイール」を提供します。

**主な強み**:
- **ネイティブなトレーニング統合:** プロダクショントレースを W&B のモデルトレーニング実行にシームレスにリンクできる唯一のプラットフォームであり、ファインチューニングのための真のデータフライホイールを実現します。
- **プログラムによる評価:** 「eval-as-code（コードとしての評価）」という強力な哲学により、硬直的な UI ベースの評価ツールよりもデータサイエンティストに好まれる、複雑でカスタムなスコアリングロジックが可能です。
- **探索的キャンバス:** ノートブックのような UI パラダイムにより、Helicone や Datadog の静的なダッシュボードと比較して、より深くアドホックなデータの探索やフィルタリングが可能です。

**改善が期待される点**:
- **エージェントの可視化:** 複雑でステートフルなエージェントループのための専用の「グラフビュー」やビジュアルデバッガーが不足しており、これは LangSmith (Studio) や Langfuse が優れている領域です。
- **ゲートウェイ/プロキシレイヤーの欠如:** Braintrust や Helicone に見られるような、キャッシュ、レート制限、フェイルオーバー・ルーティングなどのアクティブなトラフィック管理機能を提供していません。
- **人間によるアノテーションワークフロー:** LangSmith と比較して、大規模な人間によるレビューチームを管理するための成熟した UI 機能（割り当てキュー、コンセンサススコアリングなど）が不足しています。

**最新のアップデート**:
- Stats Clickhouse クエリレイヤー: 統計データを処理するためのクエリレイヤーのバックエンド改善。ダッシュボードのパフォーマンスが向上した可能性があります。(2026-01-20)
- OTel エクスポートの改善: 信頼性と標準準拠を向上させるため、OpenTelemetry のエクスポートを非同期インサートから移行しました。(2026-01-08)

### LangSmith

**概要**: LangSmith は、単純なオブザーバビリティから、フルスタックのエージェント開発、デプロイ、モニタリングソリューションへと進化した包括的な LLM エンジニアリングプラットフォームです。LangChain および LangGraph エコシステムと深く統合されており、複雑なエージェントワークフローの可視化や、ヒューマンインザループ（人間による介入）の評価プロセスの管理に優れています。フレームワークに依存しないと主張していますが、Studio のグラフビューやデプロイ機能などの最も強力な機能は、LangGraph ユーザー向けに高度に最適化されています。

**Weave に対する強み**:
- **エージェント状態の可視化:** LangGraph Studio は、Weave のトレースツリーと比較して、複雑でステートフルなエージェントループに対して優れたビジュアルデバッガーを提供します。
- **デプロイインフラストラクチャ:** LangSmith は現在、エージェントのマネージドデプロイ（「LangSmith Deployment」）を提供しており、サービングの領域までカバーしています。
- **人間によるアノテーションワークフロー:** 「アノテーションキュー」とペア比較 UI は、Weave の現在のフィードバックメカニズムよりも成熟しており、ワークフロー中心です。

**Weave に対する弱み**:
- **トレーニングとの断絶:** LangSmith はモデルのトレーニングやファインチューニングジョブとの直接的な統合がありません。Weave はオブザーバビリティを W&B のトレーニングトラッキングにシームレスに接続します。
- **フレームワークの偏り:** 技術的には依存しませんが、最高の機能（Studio、Deployment）は LangGraph に固定されています。一方、Weave は真に Python ネイティブでフレームワーク中立です。
- **モデル管理:** モデルアーティファクトのバージョニングのための専用のモデルレジストリがなく、メタデータ文字列に依存しています。

**最新のアップデート**:
- トレースプレビューのカスタマイズ: トレースの入力と出力のどの部分をトレーステーブルに直接表示するかを設定できるようになり、カスタムデータ構造の使い勝手が向上しました。(2026-02-06)
- LangSmith セルフホスト v0.13: クラウド版との機能パリティの向上、パフォーマンス、および運用制御を改善したセルフホスト顧客向けのメジャーリリース。(2026-01-16)

### Arize Phoenix

**概要**: Arize Phoenix は、OpenTelemetry および OpenInference 標準を活用して深いトレースと評価機能を提供する、オープンソースファーストの LLM オブザーバビリティプラットフォームです。ローカルファーストの開発ツールとして機能し、プロダクションモニタリングのための広範な Arize AI エンタープライズプラットフォームへとスケールします。強力な埋め込み（エンベディング）の可視化と RAG のトラブルシューティング機能で差別化を図っています。

**Weave に対する強み**:
- **オープン標準ネイティブ:** 完全に OpenTelemetry/OpenInference 上に構築されており、ベンダーロックインを避けたいチームにアピールします。
- **埋め込みの可視化:** Arize の ML の背景から継承された、ベクトル検索を可視化およびデバッグするための優れたツール（3D ポイントクラウド）を備えています。
- **ローカルファーストのワークフロー:** 初期開発にクラウド接続を必要としない、強力なローカルデバッグ体験（「Phoenix」）を提供します。

**Weave に対する弱み**:
- **トレーニング統合:** シームレスなファインチューニングループのためのトレーニングプラットフォーム (W&B) とのネイティブな接続を欠いています。
- **モデルレジストリ:** W&B に匹敵する組み込みのモデルレジストリを持たず、メタデータタグに依存しています。
- **統合プラットフォーム:** Weave はトレーニング+評価+運用を単一の画面で提供しますが、Arize は個別の ML と LLM ワークフローを橋渡しする必要があります。

**最新のアップデート**:
- Claude Opus 4.6 サポート: プレイグラウンドで Claude Opus 4.6 モデルのサポートを追加。(2026-02-09)
- ツール選択エバリュエーター: ツール選択ロジックを評価するために特化した新しいエバリュエーター。(2026-02-06)
- 忠実度（Faithfulness）エバリュエーター: より良い RAG 評価のために FaithfulnessEvaluator を追加し、HallucinationEvaluator を非推奨にしました。(2026-02-02)
- ツール呼び出し精度メトリクス: トレース内のツール呼び出しの精度を追跡するための新しいメトリクス。(2026-02-02)
- カスタムメトリクス Cursor ルール: Cursor ルールを介して新しい組み込みメトリクス（LLM 分類）を作成できる UI アップデート。(2026-01-21)

### Braintrust

**概要**: Braintrust は、評価、プロンプトエンジニアリング（プレイグラウンド）、およびオブザーバビリティを緊密に結びつけた、成熟したエンタープライズ向けの AI エンジニアリングプラットフォームです。キャッシュやレート制限のための堅牢な「AI プロキシ」、幅広い SDK サポート（Java、Go、C# を含む）、およびデータ分析用の組み込み AI アシスタント「Loop」で差別化を図っています。

**Weave に対する強み**:
- 幅広い SDK エコシステム（Java、Go、Ruby、C#）により、多様なエンタープライズスタックに対応。
- 統合された「AI プロキシ」により、キャッシュ、レート制限、コスト管理のミドルウェアを提供。
- 強力なプロンプトバージョニングと CMS 機能を備えた成熟した「プレイグラウンド」。
- 耐久性のある実行/エージェントトレースのための「Temporal」との深い統合。

**Weave に対する弱み**:
- モデルトレーニング/ファインチューニングパイプラインとの深い統合を欠いている（W&B Core とは対照的）。
- UI がより高密度で複雑であり、データサイエンティストにとって親しみやすさが劣る可能性がある。
- Weave の柔軟なオプションと比較して、大量のトレースが発生する場合の価格モデルが高価になる可能性がある。

**最新のアップデート**:
- トレースレベルのスコアラー: カスタムコードスコアラーが実行トレース全体にアクセスできるようになり、マルチステップのワークフローやエージェントの動作を評価可能になりました。(2026-02)
- LangSmith 統合: LangSmith のトレースを Braintrust にルーティングするラッパーにより、並行利用や移行が可能になりました。(2026-02)
- Cursor & MCP 統合: Cursor エディタおよび Model Context Protocol との統合により、IDE からログの照会や評価の実行が可能になりました。(2026-02)
- 自動計測 (Python/Ruby/Go): Python、Ruby、Go SDK における主要プロバイダー向けのゼロコードトレース。(2026-01)
- Temporal 統合: 分散エージェントのオブザーバビリティのための Temporal ワークフローとアクティビティの自動トレース。(2026-01)

### Langfuse

**概要**: Langfuse は、オブザーバビリティ、プロンプト管理、および評価を統合されたワークフローに組み合わせた、包括的なオープンソースの LLM エンジニアリングプラットフォームです。最近、スケール対応のためにバックエンドを ClickHouse に移行しました。堅牢なセルフホスト機能、専用のプロンプト CMS、および人間によるアノテーションやエージェントデバッグのための特殊なワークフローで差別化を図っています。

**Weave に対する強み**:
- **オープンソース & セルフホスト:** Docker を介したローカル/VPC デプロイのネイティブサポートは、セキュリティを重視するチームにアピールします。
- **プロンプト管理 CMS:** プロンプトのバージョニングとデプロイのための専用の非技術者向け UI により、プロンプトエンジニアリングをコードから分離できます。
- **アノテーションキュー:** トレースの手動レビューとラベル付けのための組み込みワークフローは、運用面でより成熟しています。
- **コストの透明性:** 多様なモデルに対する詳細で事前設定済みのコスト追跡と価格階層。

**Weave に対する弱み**:
- **トレーニング統合:** W&B Weave が継承しているモデルトレーニングの実行やアーティファクトとの深いネイティブな統合を欠いています。
- **インタラクティブな探索:** UI はダッシュボード中心であり、Weave のような柔軟でノートブック形式のデータ探索キャンバスがありません。
- **エコシステムの広さ:** 厳密に「LLM アプリ」レイヤーに焦点を当てていますが、Weave は ML ライフサイクル全体（トレーニング -> 評価 -> プロダクション）を接続します。

**最新のアップデート**:
- トレースの修正済み出力: ファインチューニングデータセットを構築するために、トレースビューで LLM 出力の改善版を直接キャプチャ。(2026-01-14)
- 単一オブザベーション評価: トレース内の個別のオブザベーションに対して評価を実行する機能をサポート。(2026-02-09)
- 思考 / 推論のレンダリング: トレース詳細におけるモデル出力の「思考」または推論部分の視覚的レンダリング。(2026-02-09)
- 組織監査ログビューアー: セキュリティとアクセスイベントを追跡するための組織レベルの監査ログ用の新しいビューアー。(2026-02-09)

### Logfire

**概要**: Logfire (Pydantic による提供) は、OpenTelemetry 上にネイティブに構築された開発者中心のオブザーバビリティプラットフォームであり、Pydantic エコシステムや Python ベースの AI エージェントとのシームレスな統合を提供します。プロダクションモニタリング、SQL ベースのトレースクエリ、および深いコードレベルの計測によるデバッグに優れていますが、Weave に見られるような体系的な評価、データセット管理、および実験ワークフローを欠いています。

**Weave に対する強み**:
- **Pydantic エコシステムの支配:** 「マジック」のような自動計測により、膨大な Pydantic ユーザーベースに対して比類のない開発者体験を提供。
- **SQL ベースの分析:** エンジニアがトレースデータに対して複雑な SQL クエリを記述でき、Weave の UI フィルタよりも高い柔軟性を提供。
- **OpenTelemetry ネイティブ:** 完全に OTel 上に構築されており、既存のプラットフォームエンジニアリングスタック (Grafana/Datadog) との統合が容易。

**Weave に対する弱み**:
- **評価ライフサイクルの欠如:** Weave のコアバリュープロポジションである トレース → データセット → 評価 → 比較 というループがありません。
- **実験ツールの不足:** プロンプトバージョニング、プレイグラウンド、またはモデルの並列比較ビューがありません。
- **Python 以外への焦点が限定的:** OTel は普遍的ですが、Logfire の独自の価値は Python/Pydantic に強く結びついています。一方、Weave はより幅広く適用可能です。

**最新のアップデート**:

- Claude SDK 計測: Anthropic Claude SDK 専用の計測を追加。(2026-01-12)

### Helicone

**概要**: Helicone は、アプリケーションと LLM プロバイダーの間にプロキシとして位置する、オープンソースの AI ゲートウェイおよびオブザーバビリティプラットフォームです。その主な価値提案はインフラレベルの制御であり、キャッシュ、ルーティング、フェイルオーバー、レート制限を提供すると同時に、入力、出力、およびコストの高精度なロギングを組み合わせたものです。

**Weave に対する強み**:
- **ゲートウェイ機能:** Weave（パッシブなオブザーバー）が提供していない、アクティブなトラフィック制御（キャッシュ、レート制限、フォールバック）を提供。
- **ゼロコード統合:** OpenAI SDK の `base_url` を変更するだけで統合できることが多く、コードの変更やデコレータを必要としません。
- **コスト管理:** 支出を追跡し、キャッシュを介して積極的にコストを削減するための優れた機能を備えています。

**Weave に対する弱み**:
- **内部トレースの欠如:** Weave のトレースツリーのようにアプリケーション内部（関数呼び出し、検索ステップ、ロジックループ）を見ることはできず、LLM の入出力のみを可視化します。
- **限定的な評価フレームワーク:** Weave に見られるような堅牢なデータセットバージョニング、回帰テスト、および CI/CD 統合を欠いています。
- **マルチモーダル/トレーニングの系譜なし:** オブザーバビリティを、広範な W&B プラットフォームのようなモデルトレーニングやファインチューニング実験に結びつけることはありません。
